# Ontologist Critique — markthisdown Specification (iteration 1)

> Reviewer: Ontologist persona
> Date: 2026-05-08
> Target: spec iteration 1 (brownfield adoption)

---

## 1. Essence

The stated goal — "a reliable, visually complete daily markdown editor with a snappy comment workflow" — is a product description, not an essence. Strip the surface features and what remains is: *a local-first annotation layer on top of plain Markdown files that preserves interoperability*. Every invariant (comments as raw HTML, no companion files, .md extension only, LLM-readable) points here. The spec's goal section should be reframed around that core, because it determines which tradeoffs are correct — not feature parity with a rendering target.

## 2. Root cause

The motivation is "formalize the spec and surface drift" — a process need, not a user need. The underlying user need (reliable daily editing with trustworthy comment placement) is real, but the spec conflates shipping fixes with specifying the system. If the fixes were shipped but the annotation-interoperability contract were broken by a future change, the root need would remain unmet. The spec should make the interoperability contract explicit as a first-class goal, not leave it implicit across the invariants list.

## 3. Prerequisites

The spec assumes:
- The regex-based SyntaxHighlighter can be extended in-place to cover tables, code blocks, file paths, and bracket tags without architectural refactoring. This is a load-bearing but unstated assumption — if the highlighter requires a rewrite, AC9–AC11 scope collapses.
- AppKit autosave (invariant 4) interacts correctly with the new `@State`-driven comment parse (AC2). If autosave triggers a document reload mid-parse, the single-parse fix may not hold. The interaction is unspecified.
- The outline view (AC12) can derive headings from the same pass used for highlighting without an independent parse. Not stated.

## 4. Hidden assumptions

- **Synchronous parse is fast enough.** The provisional invariant "comment parse is synchronous on every document.text change" assumes documents stay small. No performance floor or document-size constraint is stated; if violated, AC2's fix degrades UX rather than improving it.
- **HTML comment syntax is unambiguous in .md.** The spec assumes `<!-- -->` never appears as intentional content. No escaping or conflict rule is defined.
- **Regex is sufficient for all surface coverage.** AC9–AC11 delegate language coverage and detection heuristics to open questions, but the constraint "no third-party rendering libraries" forecloses the fallback if regex proves insufficient.

## 5. Phase dependency invariant

Phase 4 (Outline view) depends on Phase 1 and Phase 2, but AC12 has no stated dependency on comment-navigation ACs (AC8). The Phase 2 dependency appears assumed rather than derived from AC content. This is a minor coherence gap — worth confirming whether the outline view genuinely requires the sidebar-jump work or whether the dependency can be dropped to allow parallelism. No true forward-dep violation spotted.

---

**Most load-bearing concern:** The assumption that the regex-based SyntaxHighlighter is unconditionally extensible (prerequisite 1) is unverified and, if false, invalidates the scope of Phase 3 entirely without an architectural fallback.
