# Contrarian Critique — markthisdown spec iteration 1
<!-- generated 2026-05-08 -->

## Assumptions surfaced

1. The existing comment workflow is the core value; everything else serves it.
2. A machine-readable spec now is worth the time versus just shipping features.
3. Daily-use reliability requires the full markdown surface (AC9–AC11) before outline (AC12).
4. Removing the sidebar search bar (AC3) is an improvement, not a loss.
5. The "no third-party rendering libraries" constraint is stable and intentional.
6. Scroll preservation (AC16 / Invariants) is a solved, non-fragile problem.
7. Outline view (AC12) depends on AC8 (sidebar card jump) — the dependency is meaningful.

## Inverted assumptions

- **Invert 1.** If the comment workflow is already working (v2.0.12), shipping more markdown-surface polish (AC9–AC11) before solidifying comment edge cases may be the wrong priority order. The spec phases bugs-first but bundles in major new surface.
- **Invert 4.** AC3 removes sidebar search with no stated user-research rationale. If search was used, this is a regression. The only AC is negative ("search bar absent") — there is no positive replacement. What does the user do instead?
- **Invert 5.** "No third-party rendering libraries" forces AC11 (code syntax highlighting) to be hand-rolled. AC11's open question ("which engine?") shows this constraint is already in tension. The constraint may be the cause of the open question, not a protection against it.

## Challenge the problem statement

- **AC3 vs. doing nothing.** Removing search has zero stated motivation. The project survives perfectly without this delta. It should either be justified or cut.
- **AC11 vs. doing nothing.** Code-block syntax highlighting with an undefined language list and undefined engine is not ready to specify. It is an open question masquerading as an AC. Shipping it without resolving the open question produces either a trivially small feature (one language) or a scope explosion.
- **Doing nothing on spec formalization.** The code already shipped. The risk of spec/code drift is that this spec becomes stale immediately after ratification if no enforcement mechanism exists. There is none stated.

## Brownfield-specific passes

- **Invert the motivation.** The stated motivation is "surface code/spec drift." But if the code is working for daily use, the drift is academic. Is there evidence that the absence of a spec caused a concrete problem?
- **Invert the invariants.** "Scroll preservation across applyHighlighting" is both an invariant and the subject of v2.0.12's last commit. Making a freshly patched behavior an immutable invariant prevents future refactors to applyHighlighting that might be necessary for AC11 (syntax highlighting lives in the same pass).
- **Challenge AC3 trigger.** Could the sidebar search have been kept and scoped down rather than deleted? The spec does not show the removal was user-requested.

## Biggest risk

If the "no third-party rendering libraries" constraint (Constraints section) holds, AC11 has no viable engine and will either be cut after ratification or implemented as a one-language stub — making AC11 a placeholder AC that gives false confidence in iteration completeness.
