### Rough-spec claims `[from-rough-spec]`

- V2 is focused on HTML comment support (sidebar, margin icons, smart insertion) with no sigil requirement — all comments are first-class (from V2-PLAN.md)
- Selection-aware insertion behavior: empty selection → inline, selection with newline → block-above, selection without newline → inline-after (from V2-PLAN.md + conversation)
- Comments are stored as plain HTML `<!-- -->` in the document for LLM compatibility; no custom format or companion files (from conversation/001, V2-PLAN.md)
- Comment count parses on every keystroke and is fine for hundreds but would need debouncing for thousands (from DEVLOG.md)
- Sidebar search is minimally useful and should be considered for removal or making it actual in-document search (from DEVLOG.md)
- Comment edit body field doesn't auto-focus reliably when created via ⌘'; sometimes needs a click (from DEVLOG.md)
- Block comment standalone-line collapse works but cursor lands in invisible character range, viable but weird (from DEVLOG.md)
- Next priorities in order: (1) file-path/bracket-tag coloring, (2) sidebar-card-jumps-to-comment, (3) outline view, (4) frontmatter collapse, (5) tables, (6) code-block syntax highlighting (from DEVLOG.md)
- Things considered and declined: editing comments in margin popover (rejected—sidebar-only simpler), `@`-sigil for typed comments (declined—prefix conventions like TODO:/CITE: for future), custom `.mtd` file extension (declined—plain `.md` preserves AI-readability), Brew tap (discussed not built), git remote (not pushed anywhere, local only) (from DEVLOG.md)

### Ambiguities and tensions

- **Version mismatch**: Info.plist shows CFBundleShortVersionString as "1.2" (line 12) but DEVLOG.md says "v2.0.12" is shipping and claims major v2 changes (comments, sidebar, icons) are in. The code clearly has all v2.0 features; Info.plist is stale and needs updating.
- **Comment parsing overhead**: Code recomputes `MTDComment.parse(document.text)` on every text keystroke (via `onChange` on document.text in ContentView line 242) with no debounce. DEVLOG.md acknowledges this could be a problem at >500 comments but calls it "fine for hundreds." No measured threshold defined.
- **Sidebar search scope**: V2-PLAN.md says search filters comments by text content (rows 63–64), conversation suggests making it "actual document search" (conversation line 336). Code implements the former (search comment body + context line only, not whole doc). Intention unclear.
- **Auto-focus on new comment field**: DEVLOG.md line 67 lists as "worth a focused pass" but code handles it indirectly via `@FocusState` and `onChange(of: isFocused)` (ContentView.swift lines 412, 492–494). Works but only when sidebar is already visible; if sidebar opens async, timing might be loose.
- **Margin icon click behavior**: Code falls back to opening sidebar with first comment if precise hit-test fails (line 595–610). DEVLOG.md v2.0.10–11 entries describe hit-test issues and eventual success but don't document the fallback as intended design or debugging artifact.
- **Comment insertion plan for structural lines**: Code detects `#`, `-/*/+`, `>`, `\d+\.` as structural (line 860) and inserts block-above. V2-PLAN.md (lines 34–41) shows the 4-case table but doesn't specify the regex; rough spec and code agree, but exact pattern is only in source, not spec.
- **Frontmatter multi-line support**: Conversation mentions "having empty is good but also definitely good if it can be multi line in practice" (conversation line 250). Code inserts `description: |` but no collapse feature exists (DEVLOG.md line 71 lists "Frontmatter cannot be collapsed" as known issue). No spec of when/how to edit multi-line descriptions.
- **Comment edit commit semantics**: Code uses `onChange(of: bodyFocused)` to detect blur and write (line 495–501). Multiple events fire: blur → focused=false → onUpdate called. No spec of partial edits, concurrent sidebar + raw-mode editing, or undo behavior.
- **Raw mode comment display**: Code still marks comment ranges with `.mtdComment` and `.mtdCommentLocation` attributes in raw mode (lines 1144–1149) so margin icons render, but text is not hidden. No spec says whether raw mode should show markers or not, or if raw is "plaintext viewable, edit anywhere" vs "plaintext but comments still hidden."
- **Code fence exclusion**: Code searches for triple-backtick pattern with `.anchorsMatchLines` option (line 71), excluding comments inside fences from hiding/listing/icon-marking. No spec clarifies: what if fence is indented? What if backticks are in a comment (e.g., `` <!-- ` --> ``)? What if the comment delimiter itself appears as literal text in rendered code (e.g., user teaching about HTML comments)?

### Suggested axes for interview to confirm

- **Motivation**: Is v2.0.12 shipped and stable, or is this a request to lock in spec for the next iteration (v2.1 / v3)? What is the user trying to accomplish next?
- **Invariants**: Which behaviors *must not change* when iterating? (E.g., "comments stay as plain `<!-- -->`, sidebar optional, margin icons always visible," etc.)
- **Change delta**: Are we validating the existing v2.0 behavior as-is, or ratifying a *different* v2 design than what shipped? (E.g., if Info.plist should say 2.0.12, is that the only fix, or are there design changes?)
- **Scope boundary**: For next iteration, what is explicitly out of scope? (E.g., "no threaded comments," "no typed comment prefixes yet," "outline view deferred to v2.1")
- **Success criteria**: How will the user know this iteration is "done"? (E.g., "comments work smoothly for 100+ comments," "margin icons always clickable," "edit-on-blur never loses data")
- **Outstanding decisions**: Should sidebar search be narrowed to in-document search or removed? Should auto-focus on new comments be guaranteed or acceptable as-is? Should raw mode show or hide comment delimiters?
- **Performance thresholds**: At what comment count should debouncing be mandatory? How many comments per document is expected typical use?
- **File-path / tag coloring**: DEVLOG priority #1 is "file-path and `[bracket-tag]` coloring." Is this a hard requirement for next iteration, or aspirational? What defines a "file path" (regex, extension check)?
