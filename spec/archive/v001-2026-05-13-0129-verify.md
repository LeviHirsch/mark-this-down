## AC8.2 — Editor cursor is moved to comment location on card click
**Verdict:** PASS
**Evidence:** markthisdown/ContentView.swift:1019-1020 (`let r = NSRange(location: token.location, length: 0); tv.setSelectedRange(r)`). `token.location` is `MTDComment.id`, which equals `range.location` (the start of `<!--`) per :27 and :59.

## AC8.3 — Jump works whether comment is above or below current viewport
**Verdict:** PASS
**Evidence:** markthisdown/ContentView.swift:1015-1023. The jump path is direction-agnostic — `scrollRangeToVisible` is the AppKit primitive that handles either direction, and there is no conditional gating on viewport-relative position. The nonce-vs-`lastAppliedJumpNonce` comparison (:1016, :1023, :1101) ensures re-clicks of the same card jump again even when `location` is unchanged.
**Reason:** AppKit's `scrollRangeToVisible` resolves both up-scroll and down-scroll cases natively; no codepath branches on viewport position.

### Invariant — Scroll position is preserved across `applyHighlighting` calls (also AC16.1 `[adopted]`)
**Verdict:** PASS
**Evidence:** `applyHighlighting`'s save/restore at markthisdown/ContentView.swift:1198-1202 is untouched. In `updateNSView`, `applyHighlighting` runs at :1009 *before* the jump block at :1015-1023, so the highlight pass completes its save/restore first, then the jump (only when nonce changed) sets the scroll origin intentionally. Passive re-renders (keystrokes, mode toggle, selection change) leave `jumpToken.nonce` unchanged, so the jump block is a no-op and the restored scroll wins.
**Reason:** Ordering in updateNSView ensures highlight-driven re-renders never move the viewport, while sidebar-driven jumps do.
